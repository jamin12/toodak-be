# Google 로그인 + 회원 도메인 구축

## 목표
- Google ID Token 기반 모바일 로그인 구현
- 자동 가입 + JWT(Access + Refresh) 발급/회전/탈취 감지
- 멀티 디바이스 로그인 지원
- 헥사고날 아키텍처(Ports & Adapters) 기반 구조 정착

## 배경
- 신규 프로젝트 — 비즈니스 도메인이 아직 비어 있음 (`ToodakBeApplication.kt`만 존재)
- 모바일 클라이언트를 위한 서버, **소셜 로그인만 지원** (자체 이메일/비번 가입 없음)
- Google부터 시작하되 Kakao/Apple 추가 가능성 있음 — 멀티 제공자 구조로 설계
- 외부 출시 예정이라 보안 가드(자동 연결의 emailVerified 체크, 토큰 탈취 감지)는 1차부터 반영
- 표시정보(닉네임 등), 약관 동의는 별도 컨텍스트로 분리 예정 — 1차 스코프 제외

## 접근 방식
- **헥사고날 아키텍처**: 컨텍스트별로 `domain` / `application` / `adapter` 3단 분리
- **컨텍스트 분리**:
  - `member` — Member, SocialIdentity
  - `auth` — RefreshToken, Google 검증, JWT 발급/검증
  - (추후) `info` — 닉네임/표시 정보
  - (추후) `terms` — 약관 동의 이력
- **OutPort 분리 규칙**: "한 외부 의존 = 한 OutPort" — 모호함 0인 일관 규칙
- **OAuth 흐름**: ID Token Flow (모바일 친화, client_secret 불필요)
- **인증 인프라**: Spring Security 필터 체인 + JJWT

## 핵심 비즈니스 로직

### LoginWithGoogleUseCase
입력: `{ idToken, deviceId, deviceLabel? }` → 출력: `{ accessToken, refreshToken, expiresIn }`

1. **Google id_token 검증** — `GoogleOAuthOutPort.verifyIdToken(idToken)` → `GoogleUser{sub, email, emailVerified, name?, picture?}` (실패 시 `InvalidIdTokenException`)
2. **기존 SocialIdentity 조회** — `SocialIdentityOutPort.findBy(GOOGLE, sub)`
3. **분기 A (재로그인, SocialIdentity 존재)**:
   - `MemberOutPort.findById(identity.memberId)` → Member
   - `Member.status == WITHDRAWN` 이면 `ForbiddenException`
   - `Member.email != google.email` 이면 `Member.changeEmail(google.email)` → save
4. **분기 B (신규 가입, SocialIdentity 없음)**:
   - `google.emailVerified == true` 이고 `provider.trustsEmailVerification == true` 인 경우만 자동 연결 시도:
     - `MemberOutPort.findByEmail(google.email)` → 있으면 그 Member에 SocialIdentity 추가
     - 없으면 `Member.register(google.email)` → 새 Member 생성
   - emailVerified가 false면 무조건 새 Member 생성 (Google에선 발생하지 않음, Kakao 대비)
   - `SocialIdentity.link(memberId, GOOGLE, sub, emailVerifiedAt)` → save
5. **RefreshToken 발급 (디바이스 단위)**:
   - `RefreshTokenOutPort.findActiveBy(memberId, deviceId)` → 있으면 revoke
   - `RefreshToken.issue(memberId, deviceId, deviceLabel, ttl)` → save
6. **AccessToken 발급** — `JwtOutPort.issueAccessToken(memberId, ttl)`
7. 반환

### RefreshAccessTokenUseCase
입력: `{ refreshToken, deviceId }` → 출력: 동일 TokenPair

1. `RefreshTokenOutPort.findByHash(hash(refreshToken))` — null이면 `InvalidTokenException`
2. **검증**:
   - 이미 revoked → **탈취 감지**: `RefreshTokenOutPort.revokeAllByMember(token.memberId)` 후 `ForbiddenException`
   - expired → `InvalidTokenException`
   - deviceId 불일치 → `InvalidTokenException`
3. **회전**: 현재 토큰 revoke + 새 RefreshToken 발급 (`rotatedFromId = 현재.id`) + 새 AccessToken
4. 반환

### LogoutUseCase
입력: `{ refreshToken }` → idempotent
- tokenHash로 조회 → 존재하면 revoke. 없거나 이미 revoked 여도 성공 응답.

## 기술적 결정 사항
| 결정 | 선택 | 이유 |
|------|------|------|
| 인증 방식 | Google ID Token Flow | 모바일 친화, client_secret 불필요 |
| Member ID | UUID v7 (도메인이 생성) | 외부 노출 안전, 시간 정렬 가능, JPA insert 전 ID 확정 |
| 식별자 정책 | (provider, providerUserId) 전역 unique | Google sub은 영구 불변, 이메일은 변동 가능 |
| 이메일 정책 | unique 제거, 매 로그인 동기화 | 식별 키는 sub, email은 부가 정보 |
| 자동 연결 | `emailVerified=true` 제공자만 | 이메일 탈취 → 계정 가로채기 방어 |
| 탈퇴 처리 | soft delete (status WITHDRAWN) | 데이터 보존 + 어뷰징 추적 (재가입 정책은 추후 결정) |
| 세션 | JWT (Access + Refresh) | stateless, 모바일 친화 |
| Refresh 정책 | 멀티 디바이스 + 회전 + 탈취 감지 | 표준 보안 패턴 |
| 디바이스 식별 | clientId (FE가 발급한 UUID) | 디바이스 단위 세션 관리, "이 기기 로그아웃" 가능 |
| OutPort 분리 | "한 외부 의존 = 한 OutPort" | 케이스 판단 없이 자동으로 결정되는 일관 규칙 |
| 약관/Profile | 별도 컨텍스트로 분리, 1차 제외 | 책임 분리, YAGNI |
| **DB** | **PostgreSQL** 단일 (로컬/운영 모두) | 사용자 결정 — 환경 차이 최소화 |
| **JWT 라이브러리** | **jjwt** (io.jsonwebtoken) | 사용자 결정 — 단순함, Spring Security 통합 용이 |
| **의존성 위치** | 루트 `build.gradle.kts`에 직접 추가 | 사용자 결정 — 모듈 분리 시 컨벤션 플러그인으로 이동 |
| **DDL 관리** | Flyway 마이그레이션 | 운영 안전성, 버전 추적, `ddl-auto=validate` |
| **테스트 데이터** | Instancio (Kotlin variant) | 스킬 규칙 |
| **테스트 프레임워크** | Kotest BehaviorSpec + MockK | 스킬 규칙 |
| **매퍼** | JPA 매퍼는 확장함수, REST 매퍼는 MapStruct + 확장함수 공존 | 스킬 규칙 |

## 영향 범위
- **신규 패키지**: `org.toodakbe.member`, `org.toodakbe.auth`, `org.toodakbe.common`
- **신규 의존성**: spring-boot-starter-security, spring-boot-starter-validation, spring-boot-starter-data-jpa, jjwt(api/impl/jackson), postgresql, flyway-core, flyway-database-postgresql, mapstruct + mapstruct-processor(kapt), instancio-kotlin
- **신규 컨벤션 플러그인 적용**: 루트 `build.gradle.kts`에 `id("jpa")` 추가
- **신규 설정**: `application.yml`의 jwt(secret/access-ttl/refresh-ttl), datasource(PostgreSQL), google.audience(client_id)
- **DB 신규 테이블**: `member`, `social_identity`, `refresh_token` (Flyway 마이그레이션)

## 예상 결과
- `POST /api/v1/auth/google` — ID Token 받아서 가입/로그인 + TokenPair 반환
- `POST /api/v1/auth/refresh` — 회전된 TokenPair 반환
- `POST /api/v1/auth/logout` — 디바이스 토큰 폐기
- 보호된 엔드포인트는 `Authorization: Bearer <access>` 헤더로 인증
- 인증된 사용자는 `AuthenticatedMember` principal로 접근 가능
